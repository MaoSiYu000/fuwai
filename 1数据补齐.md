# 数据补齐（稳健版）说明

> 目标：在“数据有限、缺失较多”的情况下，做**保守补齐**来提升聚类可用性，且尽量避免补齐引入严重偏差。  
> 原则：**先审计、再补齐、后评估影响**；任何补齐都要留痕可回溯。

---

## 1. 目录与文件

- 代码目录：`code/data_impute/`
- 输出目录：`output/data_impute/`
- 一键运行：`python code/data_impute/run_all.py`

---

## 2. 补齐策略（稳健口径）

### 2.1 为什么不能“直接全局均值补齐”？

- 会把不同学生、不同学期的差异抹平；
- 容易把群体结构“补得太像”，影响聚类边界；
- 在高缺失特征上可能制造假信号。

### 2.2 本模块采用的分层保守补齐

对每个数值特征，缺失值按以下顺序补齐：

1. **学生历史中位数（XH）**
2. **同学期中位数（TERM_KEY）**
3. **全局中位数（兜底）**

同时做两件事：
- 为每个特征生成缺失指示列：`is_missing_*`
- 对高缺失特征打风险标记：`high_missing_feature`（默认缺失率 > 60%）

> 解释：这样做比“统一全局均值”更稳，因为优先使用“同一学生”与“同期群体”的信息。

## 3. 脚本与产出

### Step 00：缺失审计

- 脚本：`code/data_impute/00_missing_audit.py`
- 输入：`output/eda/features_student_term_draft.csv`
- 输出：
  - `output/data_impute/00_missing_audit.csv`
  - `output/data_impute/00_feature_summary.csv`

作用：
- 看每列缺失率、可数值化比例、唯一值数量；
- 判断哪些特征适合补齐，哪些要谨慎使用。

### Step 01：稳健补齐

- 脚本：`code/data_impute/01_safe_impute.py`
- 输入：`output/eda/features_student_term_draft.csv`
- 输出：
  - `output/data_impute/01_features_imputed_safe.csv`
  - `output/data_impute/01_impute_log.csv`

作用：
- 生成补齐后的可用特征表；
- 记录每个特征补了多少值、来源占比（XH/TERM_KEY/全局），便于审计。

### Step 02：补齐影响评估（关键）

- 脚本：`code/data_impute/02_impute_impact_report.py`
- 输入：
  - `output/eda/features_student_term_draft.csv`（补齐前）
  - `output/data_impute/01_features_imputed_safe.csv`（补齐后）
- 输出：
  - `output/data_impute/02_impute_impact_report.csv`

作用：
- 对每个特征比较补齐前后：均值、标准差、中位数、P90 变化率；
- `alert_large_shift=1` 表示变化过大（默认阈值 20%），需要人工复核。

---

## 4. 风险控制（避免严重后果）

1. **不直接删除原始缺失信息**：保留 `is_missing_*` 指示列，避免模型把“缺失模式”丢掉。
2. **高缺失特征不盲信**：`high_missing_feature=1` 的特征默认只做辅助解释，不作为核心决策依据。
3. **先看影响再入模**：若 `02_impute_impact_report.csv` 中某特征位移过大，优先降权或剔除。
4. **补齐是增强，不是造数据**：不做复杂生成式补样本（如 GAN 扩样），先保证稳定与可解释。

---

## 5. 下一步怎么接入聚类

建议有两条并行对照线：

- A 线（当前主线）：继续用 `cluster_prep` 现有输入；
- B 线（补齐增强线）：把 `01_features_imputed_safe.csv` 接入 `cluster_prep` 后再跑一遍；

然后比较：
- K 稳定性（ARI/NMI + 概率差异）
- 模式可解释性（mode/subtype 是否更清楚）
- 是否出现补齐导致的“大位移告警”

如果 B 线在稳定性和解释性都更好，再替换为主线。

### 6.2 补齐前后聚类对照实验（mode=8，两版同口径）

新增脚本：
- `code/data_impute/03_compare_cluster_before_after.py`

对照输出：

- `output/data_impute/compare/imputed_student_term_modes_and_subtypes.csv`
- `output/data_impute/compare/imputed_soft_k_evaluation_mode.csv`

核心结果（imputed vs baseline）：
- 关键数值速览（按你要求保留具体数值）：
  - `BIC` 变好：`62489.44 -> 56337.24`
  - 但稳定性明显变差：
    - `ARI: 0.9529 -> 0.7563`
    - `NMI: 0.8663 -> 0.7823`
    - `prob_L1: 0.01948 -> 0.06878`（越小越好，这里变差）
    - `prob_JSD: 0.02968 -> 0.16177`（越小越好，这里变差）
  - 最大簇占比：`35.70% -> 47.35%`（更容易“大簇吞并”）

- **拟合（BIC）**：更低（`62489.44 -> 56337.24`），拟合看起来更好
- **硬标签稳定性**：下降
  - `k8_ari_pair_mean: 0.9529 -> 0.7563`
  - `k8_nmi_pair_mean: 0.8663 -> 0.7823`
- **软概率稳定性**：明显变差（越小越好）
  - `k8_prob_l1_pair_mean: 0.01948 -> 0.06878`
  - `k8_prob_jsd_pair_mean: 0.02968 -> 0.16177`
- **不确定性**：
  - `mode_pmax_mean` 略升（`0.9014 -> 0.9056`）
  - `mode_entropy_mean` 略升（`0.2708 -> 0.2749`，不确定性略增）
- **簇规模均衡性**：
  - 最大簇占比上升（`35.70% -> 47.35%`），出现更明显“大簇吞并”

### 6.3 结论（是否有实质提升）

- 结论：**当前这版补齐没有带来实质提升**。  
  原因：虽然 BIC 更低，但稳定性（尤其软概率稳定性）明显恶化，且簇规模更不均衡。
- 决策：**主线暂不切换**，继续使用 baseline（当前 `mode=8` 主线口径）。
- 后续优化方向（若继续迭代补齐）：
  1. 仅对中低缺失特征补齐（高缺失列优先保留缺失指示，不强补）
  2. 对高缺失特征设置更严格补齐白名单
  3. 做“特征分组补齐”而非全量补齐，再重跑对照
